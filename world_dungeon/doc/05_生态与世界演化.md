### 5.1 生态系统（ecology_system）

**Priority：75**
**on_tick：100**（每100tick执行一次，不需要每tick都跑）

**职责**：监控种群状态，在必要时介入，避免世界陷入死寂或爆炸。

**world上的生态数据**：
```
world.ecology = {
    population = {
        [faction_id] = {
            count       : number
            peak        : number   -- 历史最高
            last_spawn  : number   -- 上次生成的tick
        }
    },
    floor_activity = {
        [floor_num] = {
            kills_this_period  : number   -- 最近一个周期的击杀数
            entity_count       : number
        }
    }
}
```

**ecology_system订阅**：
- `entity_died` → 更新对应faction的population count
- `entity_spawned` → 同上

**每100tick执行的检查**：

```
对每个楼层：
  如果 entity_count < floor_config.min_population
    → 触发补充生成（在合适的房间spawn怪物）

  如果 entity_count > floor_config.max_population
    → 触发生态压力事件（怪物开始向下迁移，或互相攻击）

  如果某个faction count == 0 且距离last_spawn超过阈值
    → 触发重生成（该faction在该层重新出现）
```

**floor_config中的生态参数**：
```lua
ecology = {
    min_population  = 5,
    max_population  = 20,
    respawn_delay   = 3000,   -- tick，全灭后多久重新出现
    spawn_per_cycle = 2,      -- 每次补充最多生成几个
}
```

### 5.2 Spawn系统（spawn_system）

**Priority：72**

**职责**：执行所有实体的生成和清理，是ecology_system和其他触发生成的系统的执行层。

**接口**：
- `request_spawn(world, def_id, room_id, overrides)` → 在指定房间生成实体，通过emit `spawn_requested`事件触发，而不是直接调用
- spawn执行后emit `entity_spawned`

**生成条件检查**：
- 目标房间是否存在
- 目标房间的实体数量是否超过上限（防止同一房间堆满）
- 目标房间类型是否允许该实体生成（比如boss房不随机刷小怪）

### 5.3 冒险者目标系统

这是整个模拟最复杂的AI部分，采用**分层目标**设计。

**adventurer_goal组件**：
```
adventurer_goal = {
    ultimate  : "retrieve_yendor"   -- 终极目标，永不改变
    
    current_stage : string
    -- "explore_floor"    正在探索当前层
    -- "seek_strength"    觉得自己不够强，在当前层刷级
    -- "find_stairs"      准备下一层
    -- "descend"          正在向楼梯移动
    -- "hunt_boss"        找到并挑战boss
    -- "retrieve_artifact" 取得目标物品
    -- "escape"           危险，逃跑
    
    target_floor  : number     -- 当前目标层数
    known_stairs  : eid | nil  -- 已知的楼梯位置
    explored_rooms : { eid → true }  -- 去过的房间
}
```

**阶段转换逻辑（在ai_system的adventurer行为原型中）**：

```
seek_strength → find_stairs 的条件：
  当前层的怪物对自己威胁度低（平均能一打三以上）
  AND 背包里有足够的补给（药水数量 > 阈值）
  AND 等级 >= floor_config.recommended_level

find_stairs → descend 的条件：
  已知楼梯位置（探索到了stairs房间）

explore_floor → seek_strength 的条件：
  当前层探索度 < 0.5（探索房间数/总房间数）
  OR 当前层有明显更强的敌人（被打到逃跑过）

escape触发条件（任何阶段都可能打断）：
  hp < hp_max * 0.25
  AND 背包里没有治疗道具
```

**威胁度评估函数**（ai/util/assessment.lua）：
```
assess_threat(world, eid, enemy_eid) → number
  综合考虑：
    双方hp比值
    双方attack/defense比值
    双方等级差
  返回 0-1，越高代表敌人越危险
```

### 5.4 冒险者的生成条件

**在ecology_system或独立的adventurer_spawner中**：

```
生成新冒险者的条件（每N tick检查一次）：
  当前活跃冒险者数量 < config.max_adventurers
  AND 距离上次生成超过 config.adventurer_spawn_interval tick
  AND 地牢入口房间实体数量未超限

生成参数：
  随机选择一个adventurer archetype定义
  level从1开始
  随机初始装备（来自archetype定义的starting_gear）
  从第一层入口生成
```

**adventurer archetype数据示例**：
```lua
-- data/entities/adventurers/warrior.lua
return {
    id   = "warrior",
    name_pool = { "Aldric", "Brynn", "Caelen" },  -- 随机取名
    
    components = {
        stats = {
            hp_max  = 40, hp = 40,
            attack  = 8,  defense = 5,
            speed   = 5,  level   = 1,
        },
        actor = {
            move_cooldown   = 25,
            attack_cooldown = 40,
        },
        faction = {
            id = "adventurer",
            hostility = {
                goblin_tribe = "aggressive",
                undead       = "aggressive",
                adventurer   = "friendly",
            }
        },
        equipment_slots = {},
        inventory = { capacity = 30, weight_current = 0, items = {} },
        skills    = {},
        level     = { current = 1, xp = 0, xp_next = 100 },
        ai_behavior = { archetype = "adventurer" },
        adventurer_goal = {
            ultimate      = "retrieve_yendor",
            current_stage = "explore_floor",
            target_floor  = 1,
            explored_rooms = {},
        },
        loot_table = {},   -- 冒险者死亡掉落身上的装备和物品
    },
    
    starting_gear = {
        -- 生成时随机给予
        guaranteed = { "iron_sword", "leather_armor" },
        random = {
            { item_id = "health_potion", count = {1,3}, chance = 0.8 },
            { item_id = "torch",         count = {2,5}, chance = 1.0 },
        }
    },
    
    growth_type = "warrior",   -- 引用level_curve里的growth配置
}
```

### 5.5 Yendor与终局

**Yendor作为一个特殊物品实体**：
```lua
-- data/items/misc/yendor.lua
return {
    id   = "amulet_of_yendor",
    name = "Yendor护符",
    tags = { "artifact", "unique", "quest_item" },
    
    components = {
        item_info = {
            weight = 0.1, value = 99999,
            stackable = false, stack_count = 1,
        },
        location  = { type = "ground" },
        unique    = { spawned = false },   -- 全局只有一个
    }
}
```

Yendor生成在最深层的boss_room，boss死亡后掉落。

**终局流程**：

冒险者拾取Yendor后：
1. `adventurer_goal.current_stage` 变为 `"escape_with_yendor"`
2. AI行为切换：不再向下，而是往第一层入口跑
3. 到达入口后emit `yendor_retrieved`事件
4. log_system记录这个冒险者进入Hall of Fame
5. `world:destroy_entity(eid)`（消亡）
6. Yendor物品实体也销毁
7. 在最深层重新生成一个新的Yendor（新的循环开始）

**Hall of Fame记录**（在world上）：
```
world.hall_of_fame = {
    {
        name        : string
        archetype   : string
        level       : number
        ticks_taken : number    -- 从生成到完成的tick数
        kills       : number
        tick        : number    -- 完成时的tick
    },
    ...
}
```
