# 地牢模拟器设计文档 v1.0

## 概述

一个无头、自驱动的地牢世界模拟器。没有玩家输入，没有图形界面，所有实体由AI驱动。唯一的输出是结构化日志。目标是构建一个能够自我演化、产生有趣叙事的模拟世界。

**技术栈**：Lua
**核心架构**：ECS（Entity-Component-System）+ Data Driven
**时间单位**：1秒 = 10 tick

---

## 第一层：核心基础设施

> 目标：世界容器能跑起来，没有任何游戏内容，但骨架完整。

### 1.2 World（ECS容器）

World是整个模拟的唯一状态容器，所有数据都在这里。

**职责**：
- 实体的创建与销毁
- 组件的增删查改
- 空间索引维护（room_id → entities）
- 事件的发布与订阅
- Tick推进

**实体管理**：
- `create_entity()` → 返回唯一integer id
- `destroy_entity(eid)` → 不立即删除，加入`_destroyed`队列，tick末统一清理，避免遍历时修改
- `is_alive(eid)` → 检查实体是否存在且未被标记销毁

**组件存储结构**：
```
_components = {
    ["position"] = {
        [eid_1] = { room_id = 42 },
        [eid_2] = { room_id = 17 },
    },
    ["stats"] = {
        [eid_1] = { hp = 30, hp_max = 30, attack = 8 },
    },
    ...
}
```
- 组件数据是纯table，无方法，所有逻辑在System里
- `add_component(eid, name, data)`
- `get_component(eid, name)` → 返回data或nil
- `remove_component(eid, name)`
- `has_component(eid, name)`

**多组件联合查询**：
- `query(comp_names[])` → 返回同时拥有所有指定组件的`{ eid → true }`
- 实现：以最小的组件集合为主表，其余做过滤，避免全量遍历

**空间索引**：
- `_room_index = { room_id → { eid → true } }`
- 这是冗余结构，source of truth是position组件
- 当position组件变化时自动同步
- `get_entities_in_room(room_id)` → 返回该房间所有实体id集合
- `move_entity_to_room(eid, new_room_id)` → 同时更新position组件和空间索引

**事件系统**：
- `subscribe(event_name, fn, system_name)` → 注册监听
- `emit(event_name, data)` → 立即分发给所有订阅者，同时追加到`event_log`
- `event_log` 是只增不减的历史记录，日志系统的数据来源

**Tick**：
- `advance_tick()` → tick+1，然后执行`flush_destroyed()`

### 1.3 Scheduler（调度器）

**职责**：管理所有System的注册与执行顺序。

**System的接口规范**：
每个System是一个table，必须包含：
```
system.name     : string   必须唯一
system.priority : number   数字小的先执行
system.update   : function(world)
```
可选：
```
system.on_tick  : number   每N个tick执行一次，默认1
system.init     : function(world)  首次注册时调用一次
```

**执行规则**：
- 按priority升序执行
- 每个system执行前检查`on_tick`，不满足则跳过
- system内部不直接调用其他system，通过emit事件通信

**建议的优先级区间**：
```
0-9   : 时间/计时类（action_timer_system）
10-29 : AI决策类（ai_system）
30-49 : 行动执行类（movement, combat, item）
50-69 : 状态更新类（status_system, stats_system）
70-89 : 生态/世界类（ecology_system, dungeon_system）
90-99 : 日志/输出类（log_system）
```

### 1.4 Registry（数据注册中心）

**职责**：所有data/目录下的定义文件在启动时注册到这里，运行时按id查询。

**存储分类**：
```
entity_def    ← 所有生物/冒险者定义
item_def      ← 所有物品定义
skill_def     ← 所有技能定义
status_def    ← 所有状态效果定义
behavior_def  ← 所有AI行为定义
room_template ← 所有房间模板定义
floor_config  ← 每层地牢的参数配置
```

**核心接口**：
- `register(category, id, def)` → 注册，id重复时报错
- `get(category, id)` → 获取，id不存在时报错
- `load_directory(path)` → 递归加载目录下所有lua文件并自动注册
- `spawn_entity(world, def_id, overrides)` → 根据定义实例化一个实体，overrides允许覆盖部分组件数据

`spawn_entity`的行为：
1. 从`entity_def`取出定义
2. 创建新entity
3. 遍历定义中的`components`，逐一`add_component`（deep copy，避免共享引用）
4. 如果有overrides，merge到对应组件
5. 添加`definition`组件记录来源id（调试用）
6. 返回eid

### 1.5 util.lua

提供以下工具函数：
- `deep_copy(t)` → 深拷贝table
- `table_merge(target, source)` → 浅merge，source覆盖target同名key
- `table_size(t)` → 计算table长度（包括非数组key）
- `table_contains(t, value)` → 线性查找
- `shuffle(t)` → 随机打乱数组（Fisher-Yates）
- `weighted_random(items)` → items是`{ value, weight }[]`，按权重随机选一个

### 1.6 main.lua

启动流程：
1. 创建World实例
2. 创建Scheduler实例
3. Registry加载所有data/目录
4. 注册所有Systems（此时systems可以做init）
5. 生成初始地牢
6. 进入tick循环

tick循环：
```
for i = 1, config.max_ticks do
    scheduler:run(world)
    world:advance_tick()
end
```

**验收标准**：
- 能执行10000次tick不报错
- 每个System按priority顺序执行
- emit的事件能被subscribe的函数接收到
- event_log正确记录所有事件
- spawn_entity能正确创建实体并挂载组件
