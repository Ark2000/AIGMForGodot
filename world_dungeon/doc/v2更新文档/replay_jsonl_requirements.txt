world_dungeon — JSONL 回放日志与外部播放器（需求说明）
================================================================================
状态：仅需求整理，实现交由后续开发（如 Claude Code）。本文档为单一事实来源。

--------------------------------------------------------------------------------
背景与目标
--------------------------------------------------------------------------------
- 现有「人类可读」日志（纯文本、语义化）保留；另增加「机器可读」JSON 日志流。
- 外部 GUI（如 LÖVE2D）作为「播放器」：读取 JSONL，按时间轴展示模拟过程。
- 协议需明确定义：每条 JSON 的对象结构、首条是否含世界初始状态、事件与 tick 的关系。

--------------------------------------------------------------------------------
1. 世界初始状态（必须在日志开头体现）
--------------------------------------------------------------------------------
- 仅追加事件流不足以还原 tick 0：地牢拓扑、房间类型、邻接关系、生成后的初始实体
  等均须在首条或专用「头记录」中给出（或与 events 配套的单文件，但需在协议中写死）。
- 播放器从「头 + 事件」即可重建用于展示的视图模型，无需再跑 Lua 生成器。

--------------------------------------------------------------------------------
2. 状态缓存 / 回放性能（播放器侧或规范侧约定）
--------------------------------------------------------------------------------
- 拖动时间轴到任意 tick 时，不应强制每次从 tick 0 全量重放（或规范允许播放器实现
  「按 tick 缓存快照 / 检查点 + 小段重放」）。
- 是否在导出端写入周期性「checkpoint」记录，或在播放器本地算缓存，由实现选型；
  需求层面：协议应支持「足够信息」让播放器可做高效 seek（例如可选 checkpoint 行类型）。

--------------------------------------------------------------------------------
3. 专用协议 / 接口（JSON 结构）
--------------------------------------------------------------------------------
- 需要独立、版本化的 schema（例如 schema_version 字段），每条日志行一个 JSON 对象
  （JSONL：一行一个完整 JSON）。
- 明确区分记录种类，例如（名称可最终实现时再定，但需在协议中枚举）：
  - header / bootstrap：种子、max_ticks、地牢图、初始实体列表等
  - event：与模拟内部事件对齐的 type + tick + data
  - （可选）checkpoint：某 tick 的压缩状态，便于 seek
- data 内字段名、类型、entity_id 含义、与房间/楼层的引用方式，均应在同目录或同文件的
  「协议说明」中写清（可与本 txt 并列一份 SCHEMA 小节或 replay_protocol_v1.txt）。

--------------------------------------------------------------------------------
4. Lua main 行为与 CLI
--------------------------------------------------------------------------------
- 默认：无新参数时，行为与当前完全一致（仅现有人类可读日志等，不破坏现有用法）。
- 带参数时：例如指定 JSONL 输出路径（具体 flag 名由实现定，如 --jsonl=out.jsonl），
  则额外写出机器可读 JSONL；人类可读输出是否同时写 stdout/文件，由实现定，但默认
  无参路径不得改变。
- 「json 参数」含义：用户期望通过命令行开关触发结构化日志输出，而非改代码才能导出。

--------------------------------------------------------------------------------
5. 播放器角色
--------------------------------------------------------------------------------
- 从「读取 JSONL + 协议解析」角度，GUI 即播放器；不依赖 world_dungeon 源码即可播放
  符合协议的导出文件。

--------------------------------------------------------------------------------
6. 实现分工（用户说明）
--------------------------------------------------------------------------------
- 本仓库主体改造（双日志、CLI、导出协议、初始状态序列化等）计划由 Claude Code 等后续
  在熟悉全库的前提下完成；本文件仅锁定需求边界，不要求 Cursor 本会话内提交实现。

--------------------------------------------------------------------------------
验收建议（供实现后自检）
--------------------------------------------------------------------------------
- 无参运行：与改造前输出/行为一致（或文档说明的故意差异为零）。
- 带参导出：生成合法 JSONL，首条含可重建 tick0 视图的信息；后续事件 tick 单调或可排序；
  播放器（或脚本）可从任意 tick T 恢复展示状态且结果与「从 0 重放到 T」一致（在协议
  定义的视图模型范围内）。
