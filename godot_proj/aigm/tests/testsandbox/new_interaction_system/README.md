# 分布式设施交互系统 (Distributed Facility Interaction System)

基于招标-投标模式的 NPC 设施交互架构。

## 核心设计理念

```
传统架构: NPC 知道所有设施的细节 (if facility is Bed: ...)
新架构:   NPC 只表达需求，设施竞标，NPC 选择最优解
```

## 架构分层

```
NPC (需求方)                      设施 (服务方)
   │                                    │
   ├── 1. 生成需求包 (NeedPackage)       │
   │   {type: "rest", urgency: 0.8}     │
   │                                    │
   ├── 2. 广播需求 ──────────────────────►│
   │                                    ├── 评估能否满足
   │                                    ├── 生成投标 (Bid)
   │                                    │   {cost, duration, score}
   │                                    │
   ├── 3. 收集所有投标 ◄─────────────────┘
   │
   ├── 4. 评分选择 (通用算法，无硬编码)
   │   score = benefit / (cost × duration × distance)
   │
   ├── 5. 发送中标通知 ─────────────────► 锁定资源
   │
   ├── 6. 移动到设施
   │
   ├── 7. 到达，签署契约 ───────────────► 接管控制权
   │                                    ├── 执行交互逻辑
   │                                    ├── 播放动画
   │                                    ├── 应用效果
   │                                    └── 完成，释放控制权
   │
   └── 8. 恢复自主状态
```

## 配置文件格式

见 `_schema.json` 和各个设施配置。

核心字段：
- `id`: 唯一标识
- `category`: 大类（rest/food/work/training/medical/social）
- `services`: 提供的服务列表，每个包含效果、成本、约束
- `physical`: 物理属性（交互范围、碰撞等）
- `ai_controller`: 可选的自定义控制器
