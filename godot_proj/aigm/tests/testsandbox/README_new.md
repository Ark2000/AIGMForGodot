# testsandbox 需求说明（重构对齐用）

> 本文档描述**沙盒要达成什么、约束是什么、还缺什么**，便于以后改版时先对齐「产品意图」再动代码。  
> 不写实现细节；具体怎么写进工程里以代码与注释为准。

---

## 1. 这是什么东西

这是一个 **2D 小世界的玩法试验场**：地图上会有多名「猫耳行走角色」，他们和玩家共用同一套规则——会饿、会累、能打架、能捡东西、能花钱。  

玩家可以 **操作其中一名** 四处走、交互、打开背包；**镜头**可以跟着某个角色看戏，也可以自己飘着看全景。  

另外还有一块 **调试能力**：方便在世界里刷东西、看状态，用来试错数值和流程。  

**边界**：这里专注「小世界 + 角色 + 设施 + 简单经济」；**大模型对话工具**在工程别处，不要求写进这份沙盒的职责里，只要别强耦合就行。

---

## 2. 玩家能体验到什么（功能需求）

### 2.1 移动与观感

- 八方向走路，在地图上有正确的 **前后遮挡**（俯视下谁在谁前面要顺眼）。
- 可选：键盘走、点地走过去（给谁开、怎么调手感是细节问题）。

### 2.2 生存感（轻量）

- **饱食度**会自己慢慢掉， everybody 一样，不论是不是玩家在操作。
- **精力**会慢慢掉；可以通过 **休息** 补回来，避免只有「饿」这一种压力。

### 2.3 物品与经济

- 地上可以掉东西；靠近可以 **拾进背包**（自动或手动，可配置）。
- 有 **钱**（用背包里的某种货币物表示即可）；买东西要够钱，卖掉东西能换钱。
- 物品要有 **名字、图标、堆叠规则**；吃的东西要能 **缓解饥饿**。

### 2.4 交互（统一的一键）

- 用一个统一的 **互动键**（例如靠近按一次）：  
  - 若附近只有 **一个** 可互动物，就直接处理；  
  - 若 **多个**（商店和箱子叠在一起、地上好几堆东西等），就 **列清单让玩家选**。
- **新增一种可互动的点**（例如以后加「灶台」「饮水机」）时，理想情况是：**不用**再去改角色脚本里一长串「如果是 A 打开 A 面板、如果是 B…」的硬编码；而是世界里的物件 **自己声明**「我现在能不能被这名角色互动、列表上显示什么字、点下去发生什么」。

### 2.5 设施类型（当前期望）

- **商店**：买卖，钱货两清。
- **储物容器**：自己和容器之间倒腾格子。
- **床**：花时间睡一觉，补精力；同一时间最好只有一个人在用（避免两人叠在同一张床上逻辑乱掉）。
- **打工点**：花时间干完活，拿一笔钱；同样适合独占会话。
- 以上设施：**玩家走 UI**，**NPC 也可以走脚本** 完成同类结果（买卖、取物、睡、拿钱），避免「只有人会用商店」。

### 2.6 战斗（轻量）

- 有 **血**；可以 **近战挥击**；能打到别人也能挨揍；死了要有合理处理（例如镜头别跟死人）。

### 2.7 镜头与操控权

- 镜头可以 **跟着某个角色**，也可以 **自由观战** 或 **轮着看**。
- 当「镜头正在跟谁」和「键盘正在操作谁」不一致时，要有清晰规则（例如自动把操作权还给镜头目标），**不要**出现「手操 A、眼看 B」还以为自己控 B 的混乱。

### 2.8 调试与试验

- 能 **看到** 跟镜对象的关键状态（生命、饱食、精力等）。
- 能 **往世界上塞测试掉落物**，减少每次手摆场景的成本。

## 3. NPC 要表现成什么样（行为需求）

### 3.1 和玩家同一套世界

- NPC **不**应该是另一套隐身数值：饿、背包、战斗规则要和玩家对齐；差别主要在 **谁在读键盘**、**谁在跑 AI**。

### 3.2 基础行为

- **闲逛**：没有急事时在一片区域内走来走去，有点「活着」的感觉。
- **环境废话**：偶尔冒一句台词（可关、可换池子），和战斗里的喊话分开配置更好。
- **打架**：被惹毛了会 **追打 / 跑路**，血少时更想跑；太久没动静要 **冷静下来** 回到日常。
- **觅食**：饿了会优先 **吃身上的** → **捡地上的** → **从箱子里掏吃的** → **有钱就去店里买**；找不到就 **四处探一探**，而不是卡死在一个逻辑上。

### 3.3 再高一层的目标（效用、而不是写死台词式 if）

- **困**了倾向于去找地方 **睡**。
- **饿**且 **没钱买吃的**、**也没有免费食物**捡、但 **店里有卖的** 时，才应该 **强烈倾向去挣钱**——这是 **推出来的结果**，不是脚本里写「每天八点必须上班」。

### 3.4 交互诚实性

- NPC 使用商店、箱子、背包时，最好走 **和玩家同一类界面或同一套数据路径**，只是 **操作节奏** 可以慢一点、步骤少一点，看起来像「真在用手点开」。

## 4. 设计原则（重构时建议守住）

1. **规则一套，身份两种**：只有「谁在输入 / 谁在决策」分开，数值与规则不拆两套。
2. **世界里的东西胜过角色里的枚举**：新设施主要靠 **摆对象 + 配置**，而不是改角色中心文件。
3. **独占资源要说清楚**：床、工位、柜台这类「一个萝卜一个坑」的，要明确 **占用与释放**，被打断（例如挨打）时要 **干净收尾**。
4. **可调试、可观测**：以后系统复杂了，仍然要能在编辑器或 Debug 里 **看见** NPC 大概在为哪件事忙。

---

## 5. 当前已知缺口（产品 / 架构债，大白话）

- **AI 逻辑还是一大坨**：觅食、打工、睡觉、打架、闲逛都缠在一起，以后加「社交点」「任务点」会越来越难维护。
- **设施还偏「一种点一种写法」**：虽然互动入口已经往「统一登记」走了，但 AI 找路、找商店、找床，仍然有点像 **按名字认路**，没有完全变成「标签世界」。
- **数值多在 Inspector 和散装参数里**：调平衡时容易东一个西一个，缺少 **一张总表** 或一层「策划可读」的配置。
- **物品表现在是代码里一大块**：改成 **数据文件** 会更适合多人改、 diff 也更清爽。
- **仓库里可能还有没接进主场景的试验脚本**：要决定删、挂进场景、还是移到示例文件夹，避免误以为「在用」其实没用。

---

## 6. 非目标（现阶段不必强求）

- 大作级别的剧情任务、多地图无缝、复杂社交网。
- 完整「模拟市民」级别的全物品互动矩阵（但 **方向**可以朝着「物件提供互动列表」演进）。
- 和生产环境服务器、存档云同步强绑定。

---

## 7. 若要做「下一版重构」，建议的成果形态

用 **话** 说清即可，不必绑定文件名：

1. **一组清晰的「世界里能发生什么」**：例如交易、储物、休息、打工、拾取……每类行为有 **共同的开始 / 结束 / 被打断** 约定。
2. **AI 分块**：日常闲逛、战斗、生存（吃喝睡钱）最好能在结构上分家，由一个小调度器决定 **此刻听谁的**。
3. **配置驱动**：权重、阈值、台词池、设施参数，尽量 **集中或可导出**，减少改一处漏十处。
4. **文档只保留这一份「要什么」+ 代码里的「怎么做」**：避免两份大文档打架。

---

## 8. 文档怎么用

- **写功能 / 改系统之前**：先对照第 2～3 节，问「这算不算扩大范围」。  
- **重构之前**：对照第 4～5、7 节，决定这一刀是 **整理结构** 还是 **改玩法**。  
- 实现细节、文件树、接口名：**以仓库现状为准**；本文件 **不**追求与代码行级同步。
