# Lua Agent（单 session + 单工具）Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在单个 `lua_agent.lua` 中实现会话状态、固定单 function 的 `tools.json`、调用 `service_kimi/kimi.py`、解析 `output.json`、沙箱执行 `run_lua`、工具循环与原子持久化，并配套能力说明文档与可离线运行的自检入口。

**Architecture:** 驱动层（本机可信）使用 `io`/`os.execute` 读写 session 目录并拉起 Python；**唯一**对外工具名为 `run_lua`，参数 `code`；用户代码在裁剪后的环境中由 `load(..., "t", env)` 加载，`debug.sethook` 做步数上限；JSON 使用嵌入的 MIT **rxi/json.lua**（整段包在立即执行函数内，不增加第二源文件）。Python 侧保持只做 HTTP 与 `output.json`，不改为本计划范围。

**Tech Stack:** Lua 5.4（仓库内 `lua54.exe`）、Python 3 + 现有 `service_kimi/kimi.py`、Moonshot OpenAI 兼容 Chat Completions（`stream: false`）。

**Spec:** `docs/superpowers/specs/2026-04-06-lua-agent-design.md`

### Task 3: 原子写文件

**Files:**
- Modify: `lua_agent.lua`

- [ ] **Step 1: 实现 `atomic_write(path, text)`**

与 `kimi.py` 的 `_write_atomic` 对齐：先写 `path .. ".tmp"`，再替换为正式名。使用 `io.open` + `write` + `close`，失败则 `error()`。

```lua
local function atomic_write(path, text)
  local tmp = path .. ".tmp"
  local f, err = io.open(tmp, "wb")
  if not f then error(err) end
  assert(f:write(text))
  assert(f:close())
  os.remove(path)
  local ok, ren = os.rename(tmp, path)
  if not ok then
    -- Windows 上 rename 覆盖行为依实现而定；若失败可先删目标再 rename
    os.remove(path)
    assert(os.rename(tmp, path))
  end
end
```

### Task 9: 沙箱执行 `run_lua`

**Files:**
- Modify: `lua_agent.lua`

- [ ] **Step 1: 构建空环境并注入安全全局**

```lua
local function make_sandbox_env()
  local print_buf = {}
  local env = {
    math = math,
    string = string,
    table = table,
    utf8 = utf8,
    print = function(...)
      local t = { ... }
      for i = 1, #t do t[i] = tostring(t[i]) end
      table.insert(print_buf, table.concat(t, "\t"))
    end,
  }
  env._G = env
  return env, print_buf
end
```

- [ ] **Step 2: `sandbox_run(code, config)`**

- `load(code, "@sandbox", "t", env)`；失败则返回 `"[compile] " .. err`。
- 成功则 `debug.sethook(function() error("SANDBOX_TIMEOUT") end, "", config.sandbox_max_instructions)` 在调用前设置，调用后 `debug.sethook(nil)`。
- `local ok, res = pcall(f)`；若 `ok` 且 `res~=nil`，返回 `tostring(res)`；若 `ok` 且 `res==nil`，若有 `print_buf` 拼接则返回拼接，否则返回 `""`。
- 任意失败：`return "[error] " .. tostring(res)`（不把错误吞掉）。


### Task 10: 工具循环与 `llm_round`

**Files:**
- Modify: `lua_agent.lua`

- [ ] **Step 1: `extract_tool_calls(msg)`**

若 `msg.tool_calls` 为数组，返回之；否则返回 `{}`。

- [ ] **Step 2: 校验名称**

对每个 call，若 `call.function.name ~= config.tool_name`，`append_tool_result` 仍追加一条 tool，`content` 为 `"[error] unexpected tool " .. name`（spec：每条调用都要有 tool 消息）。

- [ ] **Step 3: 对 `run_lua` 解析 `arguments`**

`local args = json.decode(call.function.arguments)`，取 `args.code`；若缺字段，`content` 为可读错误。

- [ ] **Step 4: `llm_round(messages, config)`**

1. `save_messages`  
2. `local out, err = call_kimi(config)`；若失败，返回 `nil, err`  
3. `append_assistant_from_completion`  
4. `save_messages`  
5. 若 `finish_reason == "tool_calls"`：对每个 tool 执行 Step 2–3，`append_tool_result`，`save_messages`，若未达到 `max_tool_rounds`，递归或循环再次 `call_kimi`（注意：OpenAI 流程是 **写完所有 tool 消息后再请求下一轮**）。  
6. 若 `finish_reason == "stop"`（或模型返回无 tool 的结束），结束。  
7. 循环计数：仅统计「调用 Kimi 的次数」或「tool 轮数」，在计划中统一为：**连续 tool 轮** 不超过 `max_tool_rounds`（每次完成一批 tool_calls 再调 Kimi 算一轮）。

伪代码形状：

```lua
local rounds = 0
while true do
  rounds = rounds + 1
  if rounds > config.max_tool_rounds then return nil, "max_tool_rounds exceeded" end
  local out, err = call_kimi(config)
  ...
  if finish ~= "tool_calls" then break end
  -- append tools, save, loop
end
```
