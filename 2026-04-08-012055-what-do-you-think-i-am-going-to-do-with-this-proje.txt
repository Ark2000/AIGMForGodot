в—Ҹ Good direction. This is essentially a perception problem вҖ” the agent needs to
  build a mental model of the world from text, without being overwhelmed by
  noise.

  Key tensions to design around:

  Context window is finite, logs are not.
  You can't dump raw logs into every LLM call. You need a compression strategy.

  Ideas:

  1. Structured event format (not free-text logs)
  Instead of "Player dealt 14 damage to Goblin", emit JSON events:
  {type:"combat", actor:"player", target:"goblin", action:"attack", dmg:14}. The
   agent can filter, aggregate, and reason about structured data far better.
  2. World model as sandbox state
  The agent maintains a persistent Lua table (backed by a session JSON file)
  representing current game state вҖ” player HP, location, active NPCs, quest
  flags, etc. Each call it updates the model from new events, not all events.
  This is the key insight: the agent reads logs to update its model, not to 
  answer questions directly from raw logs.
  3. Log chunking with a "since last read" cursor
  Game appends to a rolling log file. Agent tracks a byte/line offset. Each call
   only processes new events since last checkpoint. Prevents re-reading the
  entire history.
  4. Tiered summarization
    - Hot buffer: last ~50 raw events (always in context)
    - Warm summary: agent-written prose summary of the last ~10 minutes (updated
   periodically via a run_lua call)
    - Cold archive: old events dropped or stored externally
  5. Importance scoring / event filtering
  Not all logs matter equally. Player death > player walked 2 steps. Either the
  game tags events with priority, or the agent filters by type before
  processing.

  ---
  Concrete next step I'd suggest:

  Design the event schema and add two sandbox APIs:
  - log_read(n) вҖ” returns last N unread structured events
  - state_get(key) / state_set(key, val) вҖ” agent's persistent world model

  These two together let the agent do: read new events вҶ’ update world model вҶ’ 
  reason вҶ’ act. That's the whole perception loop.

в—Ҹ Yes, exactly. A purely reactive agent that only responds when poked is weak вҖ”
  a good GM has agency. It notices things, plans ahead, and acts without being
  asked.

  There are really two distinct awakening mechanisms:

  1. Event triggers (reactive, but selective)
  Not every log wakes the agent вҖ” only ones it's subscribed to. The agent itself
   registers what it cares about:
  - "player_hp_critical" вҶ’ wake immediately
  - "boss_spawned" вҶ’ wake immediately
  - "player_walked_tile" вҶ’ ignore

  Crucially, the agent can change its subscriptions over time based on what it's
   doing. If it's orchestrating a boss fight, it subscribes to fine-grained
  combat events. In downtime, it only watches for major events.

  2. Scheduled routines (proactive)
  The agent sets its own alarms:
  - "Wake me in 5 game-minutes to survey the situation"
  - "Check if the player has made progress on this quest in 10 minutes, if not,
  nudge"

  This is the subjective initiative part вҖ” the agent has a pending agenda, not
  just a response queue.

  ---
  The architectural piece this implies: a Dispatcher

  Something outside the agent needs to:
  1. Receive events from the game
  2. Check them against the agent's registered triggers
  3. Wake the agent (call lua_agent.lua) with context: why it was woken and what
   happened

  The agent, when woken, always knows its wake reason: {reason: "trigger", 
  event: "player_hp_critical"} vs {reason: "schedule", name: "world_survey"}.
  That shapes how it responds.

  The agent's persistent state would carry:
  triggers: { player_hp_critical: true, boss_spawned: true }
  schedule: [ {at: 1200, name: "quest_check"}, ... ]
  agenda:   [ "pressure player toward the dungeon", "don't let them backtrack" ]

  ---
  The interesting design question: does the dispatcher live in Lua, or is it the
   game engine (eventually Godot)?

  For now, a standalone Lua dispatcher script that polls a log file and manages
  a schedule would let you develop and test all of this without Godot. Godot
  eventually becomes the dispatcher.

     Data Files (created at runtime, not committed)

     }

     session_dir/dispatcher_cursor.json

     Single number: the line offset in events.jsonl the dispatcher has already
     processed.
     42

     ---
     Sandbox API (3 additions to sandbox.lua)

     state_get(key)        -- returns value from agent_state.json[key], or nil
     state_set(key, val)   -- writes agent_state.json[key] = val (atomic)
     events_read(n)        -- returns up to n unread events as a JSON string;
                           -- advances dispatcher_cursor

     These are injected into make_env(). They close over the paths from cfg:
     - cfg.agent_state_path = session_dir/agent_state.json
     - cfg.events_path      = events.jsonl (configurable, defaults to
     "events.jsonl")
     - cfg.cursor_path      = session_dir/dispatcher_cursor.json

     The sandbox.run(code, cfg) signature is unchanged; cfg just gets three new
     optional fields.

     ---
     dispatcher.lua вҖ” Poll Loop

     Single script. Runs continuously. No daemon framework needed.

     config: session_dir, events_path, poll_interval_ms (default 1000)

     loop:
       1. read new lines from events.jsonl since cursor_path offset
       2. load agent_state.json вҶ’ triggers[], schedule[]
       3. for each new event:
            if event.type is in triggers вҶ’ wake agent, reason=trigger,
     payload=event
       4. for each schedule entry:
            if entry.at <= now вҶ’ wake agent, reason=schedule, payload=entry;
     remove entry from state
       5. sleep poll_interval_ms
       6. repeat

     Wake = shell out to lua_agent.lua with a synthetic user message (JSON
     string):
     {"wake": "trigger", "event": {"t":..., "type":"player_hp_critical",
     "data":{...}}}
     {"wake": "schedule", "reason": "world_survey"}

     The agent reads this as its "user" message at the start of main(). The
     system prompt (agent_lua_ref.md) tells it what to do with it.
